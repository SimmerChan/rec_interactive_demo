# 昇腾推荐系统可视化演示 - 创意构思

**生成日期**: 2026-05-12
**主题**: 基于昇腾平台的推荐系统可视化演示快速实现方案

---

## 背景与约束

### 已知信息

1. **Ascend 仓库能力**:
   - `RecSDK`: 昇腾推荐系统SDK，提供训练和推理基础能力
   - `fbgemm-ascend`: FBGEMM量化库昇腾适配，支持GPU/NPU量化推理
   - `HierarchicalKV-ascend`: 分层KV缓存优化，用于加速推荐系统推理

2. **NVIDIA/recsys-examples 参考实现**:
   - **HSTU 算法**: 分层语义标记统一模型，支持大规模排序和检索
   - **DynamicEmb**: 动态嵌入表，支持并行、零冲突哈希、LRU/LFU驱逐策略
   - **语义ID检索**: 基于TorchRec和Megatron-Core的分布式训练
   - **推理优化**: Triton推理服务器、CUDA图、C++ AOTInductor部署、FP8量化
   - **数据**: 支持标准推荐系统数据集格式

3. **目标流程**:
   ```
   recsys-examples算法/数据 → Ascend训练 → PyTorch推理部署 → 可视化网页
   ```

### 快速演示的核心约束

- **时间约束**: 简单快速实现，需要控制复杂度
- **可视化优先**: 演示效果 > 算法精度
- **昇腾能力展示**: 需要体现昇腾平台特性
- **交互性**: 用户能直观感受推荐效果

## 创意构思候选列表

### 1. 极简流水线架构 (Pain and Friction)

**思路**: 最小化组件数量，只实现绝对必要的部分

- 用单个Python脚本串联所有环节：数据加载 → Ascend训练 → PyTorch推理 → Flask/FastAPI服务 → 前端展示
- 放弃分布式训练，用单节点PyTorch+Ascend插件做演示
- 放弃实时推理，用预计算+缓存策略

**为什么重要**: 降低集成复杂度，确保演示当天能跑通

** warrant**: `direct:` 用户明确要求"简单快速"，最小化组件是减少风险的最直接方式

---

### 2. 嵌入空间可视化 + 实时推荐 (Inversion/Removal)

**思路**: 把"训练后推理"改为"边推荐边可视化"

- 用户选择一个物品，系统实时返回Top-K推荐
- 同时在2D/3D UMAP/t-SNE可视化中显示推荐物品与用户历史物品的嵌入空间关系
- 让用户能直观理解"为什么推荐这个"

**为什么重要**: 传统推荐系统对用户是黑箱，可视化嵌入空间是展示模型理解力的有力方式

**warrant**: `external:` recsys-examples的DynamicEmb本身就支持嵌入表操作；嵌入可视化是推荐系统论文常用的解释性方法

---

### 3. MovieLens微服务演示 (Assumption-breaking)

**思路**: 用MovieLens数据集替代真实工业数据，大幅降低数据准备难度

- MovieLens 1M数据集足够做演示
- 预训练好模型，演示时只做推理
- 用户选择电影，系统返回相似电影推荐
- 历史上有大量推荐系统论文用MovieLens做基准

**为什么重要**: 数据准备往往是企业演示的最大障碍；MovieLens是公开可用、最广泛研究的数据集

**warrant**: `direct:` 用户提到"参考recsys-examples"，而recsys-examples本身支持标准数据集格式；MovieLens是其常用的基准数据

---

### 4. 规则引擎降级策略 (Leverage/Compounding)

**思路**: 当模型推理失败时，用基于规则的简单推荐作为降级

- 优先尝试模型推荐
- 模型不可用/超时/置信度低时，自动切换到"同类别Top10"、"热门物品"、"协同过滤"等规则
- 保证演示always-on

**为什么重要**: 演示环境不稳定时，降级策略能保证演示顺利进行

**warrant**: `reasoned:` 工业推荐系统普遍采用多路召回+规则融合的级联架构；这个设计符合工程最佳实践

---

### 5. Ascend NPU量化推理优先 (Leverage/Compounding)

**思路**: 突出昇腾NPU的量化加速能力，而非训练能力

- 使用fbgemm-ascend的INT8/FP8量化推理
- 对比CPU vs NPU推理延迟
- 量化模型体积缩小展示

**为什么重要**: 昇腾NPU的量化能力是其差异化优势；量化推理比训练更易演示效果

**warrant**: `external:` fbgemm-ascend明确宣传量化加速能力；recsys-examples v26.03也重点宣传FP8量化推理支持

---

### 6. 实时反馈闭环演示 (Cross-domain Analogy)

**思路**: 借鉴在线学习系统思想，展示"推荐-反馈-更新"的实时循环

- 用户点击/跳过推荐结果
- 模拟记录反馈（不需要真实在线学习）
- 展示反馈数据如何影响下一轮推荐

**为什么重要**: 让演示不仅是静态展示，而是有交互的生命体

**warrant:** `reasoned:` recsys-examples的HSTU支持序列训练；实时反馈是工业推荐系统的核心特性

---

### 7. 嵌入式notebook集成 (Constraint-flipping)

**思路**: 把演示嵌入到Jupyter notebook中，而非独立网页

- 用户在notebook中交互
- 单元格直接输出可视化图表
- 降低部署复杂度

**为什么重要**: 如果目标用户是数据科学家，notebook比Web应用更自然

**warrant**: `reasoned:` 机器学习社区普遍使用notebook；recsys-examples本身就是notebook格式的示例

---

### 8. Docker一键部署 (Automation)

**思路**: 整个环境打包成Docker镜像，用户开箱即用

- 镜像包含: Ascend驱动、PyTorch、训练好的模型、前端服务
- `docker run` 启动所有服务
- docker-compose编排各组件

**为什么重要**: 减少用户环境配置的负担；确保演示环境一致性

**warrant**: `direct:` 用户要求"快速实现"；docker化是快速交付的标准手段

---

### 9. 推荐多样性对比展示 (Assumption-breaking)

**思路**: 对比"精准推荐"vs"多样性推荐"的不同效果

- 同一用户，分别展示"只推同类"和"跨类探索"的推荐结果
- 用户选择后展示满意度投票
- 展示推荐系统中EE(Exploration-Exploitation)的经典权衡

**为什么重要**: 推荐多样性是工业界的重要话题；这个对比能引发观众讨论

**warrant**: `external:` recsys-examples的HSTU支持多样性感知的训练；推荐多样性是推荐系统论文的常见主题

---

### 10. 预计算+增量更新混合架构 (Leverage/Compounding)

**思路**: 离线预计算大部分推荐结果，运行时增量更新

- 每天/每小时预计算热门用户的推荐列表
- 运行时快速返回，不依赖实时推理延迟
- 当用户是新用户时，走实时推理

**为什么重要**: 大幅降低演示时延，提升用户体验

**warrant**: `reasoned:` 工业推荐系统普遍采用多级缓存+预计算的级联架构；这个设计降低了对推理延迟的严格要求

## 优先建议

### 方案A (推荐): 快速演示优先

```
MovieLens数据 + 方案1(极简流水线) + 方案8(Docker) + 方案4(降级策略) + Gradio前端
```

**理由**:
1. 能在最短时间内实现端到端流程
2. Docker化确保环境一致
3. 降级策略保证演示稳定
4. Gradio开发速度快，推荐系统专用组件丰富，适合快速原型演示
5. 后续可迭代加入可视化(方案2)和量化对比(方案5)

### 方案B (备选): 效果演示优先

```
MovieLens数据 + 方案2(嵌入可视化) + 方案5(NPU量化) + 方案8(Docker) + Gradio前端
```

**理由**:
1. 更能展示昇腾平台的技术差异
2. 可视化效果更容易打动观众
3. Gradio内置丰富图表组件，嵌入可视化实现更快
4. 需要更多时间调试，但演示效果更好

## 技术架构草案

```
┌─────────────────┐     ┌──────────────────┐     ┌─────────────────┐
│  recsys-examples │────▶│   Ascend训练      │────▶│  PyTorch模型    │
│  (HSTU/DynamicEmb)│     │   (RecSDK+插件)    │     │  (原生模型)      │
└─────────────────┘     └──────────────────┘     └─────────────────┘
                                                          │
                                                          ▼
┌─────────────────┐     ┌──────────────────┐     ┌─────────────────┐
│  Gradio前端      │◀────│  FastAPI/Flask   │◀────│  NPU推理服务     │
│  (交互式推荐UI)  │     │  (推荐API)        │     │  (原生PyTorch)  │
└─────────────────┘     └──────────────────┘     └─────────────────┘
```

**技术栈详情**:
- 数据集: MovieLens 1M
- 训练框架: Ascend RecSDK + fbgemm-ascend
- 推理引擎: **原生PyTorch** (昇腾NPU后端，无需ONNX转换)
- API服务: FastAPI
- 前端: **Gradio** (推荐系统专用交互组件)
- 部署: Docker + docker-compose
